# «ьё» — рабочее понимание задачи

## Что это

«ьё» — мобильная веб-галерея, которая ощущается не как каталог и не как лента, а как цифровая колода таро. Пользователь видит одну активную карту-работу и чувствует, что под ней есть глубина: стопка, жест, переворот, смена атмосферы.

Основной образ:

- черный фон с легким шумом;
- в центре стопка карточек-изображений;
- верхняя карта полностью видна;
- остальные читаются как скрытая толща под ней;
- свайп влево или вправо переключает работу;
- тап переворачивает карту;
- на обороте: промпт, метаданные, кнопка «слушать»;
- интерфейс почти исчезает, остаются только карточки, логотип, поиск и фон.

## Как должен ощущаться интерфейс

Это не UI в обычном смысле, а маленький ритуал просмотра.

Ключевые свойства:

- минимализм без пустоты;
- тактильность через анимации, тени, параллакс, стек;
- ощущение «предсказания» или вытягивания следующей карты;
- камерность и тишина;
- неоновый акцент только там, где он действительно нужен: в логотипе и деликатно в активных состояниях.

Важно не скатиться в обычную мобильную галерею с карточками и кнопками. Центральная метафора здесь сильнее привычного UX-паттерна.

## Визуальная система

### 1. Фон

- Базовый фон: почти черный, не абсолютно плоский.
- Поверх: очень тонкий шум или зерно.
- Под активной картой: мягкое цветное свечение, вычисляемое из доминирующих цветов изображения.
- При смене карты фон должен плавно перетекать, а не переключаться резко.

### 2. Логотип

Логотип «ьё» расположен сверху, как единственный постоянный знак интерфейса.

- `ь` — полупрозрачный, мягкий, почти призрачный;
- `ё` — светится неоновым цианом;
- вся надпись компактная, с большим количеством воздуха вокруг;
- логотип не должен спорить с картой.

### 3. Карта

Лицевая сторона:

- крупное изображение;
- скругление углов не слишком большое;
- глубокая, мягкая тень;
- возможно легкое отражение света сверху.

Оборот:

- темная поверхность с типографикой;
- блок с промптом;
- метаданные: автор, серия, теги, дата, техника или модель;
- кнопка `слушать`.

Стопка:

- 2–4 нижних карты видны частично;
- каждая следующая немного смещена и уменьшена;
- стопка должна создавать ощущение физического объекта, а не списка.

## Поведение

### 1. Свайп

Свайп влево или вправо:

- запускает движение верхней карты;
- следующая карта поднимается из стопки;
- переход мягкий, но быстрый;
- жест должен чувствоваться естественно на телефоне;
- нужен и тач, и мышь/трекпад как fallback для отладки.

### 2. Переворот

Тап по карте:

- переворачивает активную карту по оси `Y`;
- лицевая и обратная стороны должны быть визуально разными;
- анимация не цирковая, а спокойная и уверенная;
- во время переворота стек не должен ломаться.

### 3. Поиск

Поиск не буквальный, а «по смыслу».

Пользователь вводит:

- настроение;
- сюжет;
- образ;
- ассоциацию.

Система должна подавать релевантные работы так, будто колода «поняла запрос».

На первом этапе это можно реализовать упрощенно:

- поле ввода;
- ручные теги, описания и ключевые слова у каждой работы;
- локальный scoring по совпадениям слов и синонимических групп;
- сортировка колоды по релевантности;
- если запрос пустой, показываем обычный порядок.

Позже можно заменить на embeddings или API-семантику.

### 4. Озвучка

Кнопка `слушать` запускает голос ИИ-куратора.

Для MVP:

- использовать Web Speech API, если доступен браузером;
- собирать короткий текст-описание из метаданных и кураторского описания;
- предусмотреть состояния `play / stop / unsupported`.

В будущем:

- подключить внешний TTS;
- выбрать фиксированный голос;
- добавить предзаписанную интонационную модель бренда.

## Что нужно в MVP

Минимальный, но уже выразительный прототип должен включать:

1. Одну HTML-страницу без сложной сборки.
2. Датасет работ в JS-массиве.
3. Мобильный fullscreen layout.
4. Стопку карт с активной верхней картой.
5. Свайп жестом и кнопочный fallback при необходимости.
6. Переворот карты.
7. Оборот с промптом и метаданными.
8. Кнопку `слушать` на базе Web Speech API.
9. Поиск с локальной семантической эвристикой.
10. Плавную адаптацию фонового свечения под активную работу.

## Предлагаемая структура проекта

Если делать на чистом `HTML + CSS + JS`, достаточно такой структуры:

```text
/
  index.html
  styles.css
  app.js
  assets/
    images/
```

Если позже понадобится масштабирование:

- `data.js` для работ;
- `search.js` для смыслового поиска;
- `audio.js` для TTS;
- `deck.js` для логики колоды;
- `theme.js` для фоновых градиентов и цветовой адаптации.

## Состояния интерфейса

Нужно продумать следующие состояния:

- загрузка первой карты;
- активная карта спереди;
- активная карта с оборотом;
- переход между картами;
- активный поиск;
- пустой результат поиска;
- воспроизведение озвучки;
- браузер без поддержки TTS.

## Основные технические риски

### 1. Свайп + переворот

Нужно аккуратно развести два жеста:

- горизонтальный drag должен листать;
- обычный tap должен переворачивать;
- важно исключить ложные срабатывания.

### 2. Производительность на мобильных

Если переборщить:

- с blur;
- с большими тенями;
- с шумом;
- с частыми пересчетами цветов,

то интерфейс станет тяжелым.

Нужен упор на:

- `transform` и `opacity`;
- умеренное число слоев;
- предсказуемую анимацию;
- кэширование цветовых данных карточек.

### 3. Семантический поиск

Без API настоящий semantic search не получится, поэтому надо честно заложить промежуточный слой: иллюзию семантики через словари настроений, образов и тегов.

## Художественные ориентиры реализации

При разработке важно держать такие рамки:

- не делать белых плашек, тяжелых инпутов и стандартных мобильных controls;
- не превращать оборот карты в обычную карточку товара;
- не перегружать экран текстом;
- избегать «дрибббл-глянца» и декоративного шума ради шума;
- помнить, что главный объект всегда один: текущая карта.

## Предлагаемый порядок сборки

### Этап 1. Каркас

- fullscreen layout;
- логотип;
- контейнер колоды;
- одна тестовая карта;
- базовая атмосфера фона.

### Этап 2. Колода

- массив данных;
- стек нижних карт;
- переключение активной карты;
- жесты и анимации.

### Этап 3. Переворот

- front/back;
- метаданные;
- промпт;
- кнопка озвучки.

### Этап 4. Атмосфера

- шум;
- цветовая адаптация фона;
- полировка теней, свечения и ритма анимации.

### Этап 5. Поиск

- поле ввода;
- локальный scoring;
- перестройка порядка колоды под запрос.

## Решения, которые сейчас выглядят разумно

- Делать сначала чистый фронтенд без фреймворка.
- Собрать выразительный MVP до подключения реального backend/search/TTS.
- Хранить все работы в локальном JS-объекте.
- Делать mobile-first, но оставить desktop-совместимость для демонстрации.
- Сразу заложить художественную целостность: типографика, темп анимации, глубина стопки и свет важнее количества функций.

## Что считать хорошим результатом первой версии

Первая версия удачна, если:

- за 3–5 секунд понятна метафора «цифровой колоды»;
- перелистывание хочется повторять;
- переворот ощущается естественно;
- текст на обороте не ломает магию;
- поиск не выглядит техничным;
- фон живет вместе с работой;
- интерфейс остается тихим и уверенным.
